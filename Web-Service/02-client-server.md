# 클라이언트와 서버

## 1. 클라이언트와 서버의 개념

웹 서비스는 클라이언트와 서버가 데이터를 주고받으며 동작한다.

| 구분 | 설명 |
|---|---|
| 클라이언트 | 서비스를 요청하는 쪽 |
| 서버 | 클라이언트의 요청을 처리하고 응답하는 쪽 |

웹 서비스에서 클라이언트는 일반적으로 사용자의 브라우저를 의미한다.  
서버는 클라이언트의 요청을 받아 처리하고, 필요한 데이터를 응답으로 반환한다.

클라이언트와 서버의 관계는 다음과 같이 정리할 수 있다.

```text
클라이언트 → 요청(Request) → 서버
클라이언트 ← 응답(Response) ← 서버
```

즉, 웹 서비스는 클라이언트가 요청을 보내고 서버가 응답을 반환하는 구조를 기반으로 동작한다.

## 2. 브라우저와 URL

사용자는 웹 서비스에 접근할 때 브라우저를 사용한다.

브라우저 주소창에 다음과 같은 URL을 입력하면 해당 웹 서비스에 접속할 수 있다.

```text
www.naver.com
www.google.com
```

URL은 사용자가 웹 서비스에 접근하기 위해 사용하는 주소이다.

실제 서버는 IP 주소를 기반으로 구분된다.  
하지만 사용자가 IP 주소를 직접 외워서 접속하기는 어렵기 때문에, 사람이 이해하기 쉬운 도메인 이름을 사용한다.

예를 들어 사용자는 복잡한 IP 주소 대신 `www.naver.com`과 같은 도메인 주소를 입력해 서비스에 접속한다.

## 3. 도메인과 IP

서버는 고유한 IP 주소를 가진다.  
IP 주소는 네트워크상에서 특정 컴퓨터나 서버를 찾기 위한 주소이다.

하지만 IP 주소는 사람이 기억하기 어렵기 때문에 도메인 이름을 사용한다.

| 구분 | 설명 |
|---|---|
| IP 주소 | 서버를 식별하기 위한 실제 네트워크 주소 |
| 도메인 | 사람이 이해하기 쉬운 서버 주소 이름 |
| DNS | 도메인 이름을 IP 주소로 변환해주는 시스템 |

예를 들어 사용자가 브라우저에 `www.naver.com`을 입력하면, DNS를 통해 해당 도메인에 연결된 IP 주소를 찾고, 그 IP 주소를 가진 서버에 요청을 보내게 된다.

정리하면 다음과 같다.

```text
사용자 URL 입력
→ DNS가 도메인을 IP 주소로 변환
→ 해당 IP를 가진 서버로 요청 전달
```

## 4. HTTP와 HTTPS

웹에서 클라이언트와 서버는 HTTP 또는 HTTPS라는 프로토콜을 사용해 통신한다.

| 프로토콜 | 설명 |
|---|---|
| HTTP | 웹에서 데이터를 주고받기 위한 기본 통신 규칙 |
| HTTPS | HTTP에 보안 기능이 추가된 통신 방식 |

HTTP는 HyperText Transfer Protocol의 약자이다.  
클라이언트와 서버가 웹에서 데이터를 주고받기 위한 약속이다.

HTTPS는 HTTP에 보안 기능이 추가된 방식이다.  
데이터를 암호화하여 전송하기 때문에 HTTP보다 안전하다.

일반적으로 로그인, 결제, 개인정보 처리와 같이 보안이 필요한 서비스에서는 HTTPS를 사용한다.

## 5. 웹 서비스의 요청 흐름

사용자가 브라우저에 URL을 입력하면 다음과 같은 흐름이 발생한다.

```text
사용자
→ 브라우저에 URL 입력
→ HTTP 또는 HTTPS 요청 전송
→ 서버가 요청 수신
→ 서버에서 요청 처리
→ 응답 반환
→ 브라우저가 응답 결과를 화면에 표시
```

웹 서비스는 이처럼 클라이언트의 요청과 서버의 응답을 중심으로 동작한다.

예를 들어 사용자가 로그인 버튼을 클릭하면 다음과 같은 흐름이 발생한다.

```text
로그인 버튼 클릭
→ 로그인 요청 전송
→ 서버에서 아이디와 비밀번호 확인
→ 데이터베이스에서 사용자 정보 조회
→ 로그인 성공 또는 실패 응답 반환
→ 브라우저에 결과 표시
```

## 6. 웹 서버

웹 서버는 클라이언트의 요청을 받아 정적 리소스를 처리하는 서버이다.

정적 리소스란 요청할 때마다 내용이 바뀌지 않는 파일을 의미한다.

예시는 다음과 같다.

- HTML
- CSS
- JavaScript
- 이미지 파일

대표적인 웹 서버는 다음과 같다.

- Apache
- Nginx

웹 서버는 클라이언트의 요청을 먼저 받고, 정적 리소스를 직접 응답하거나 필요한 경우 WAS로 요청을 전달한다.

## 7. WAS

WAS는 Web Application Server의 약자이다.  
동적 리소스를 처리하는 웹 애플리케이션 서버이다.

동적 리소스란 사용자의 요청이나 데이터에 따라 결과가 달라지는 프로그램을 의미한다.

예시는 다음과 같다.

- JSP
- Servlet
- Spring 애플리케이션
- Spring Boot 애플리케이션

WAS는 사용자의 요청에 따라 프로그램을 실행하고, 필요한 로직을 처리한 뒤 결과를 반환한다.

대표적인 WAS는 다음과 같다.

- Tomcat
- WebLogic
- WebSphere
- JEUS

Java 기반 웹 애플리케이션에서는 Tomcat이 자주 사용된다.

## 8. 웹 서버와 WAS의 차이

웹 서버와 WAS의 차이는 다음과 같이 정리할 수 있다.

| 구분 | 웹 서버 | WAS |
|---|---|---|
| 담당 리소스 | 정적 리소스 | 동적 리소스 |
| 주요 역할 | HTML, CSS, JS, 이미지 제공 | 비즈니스 로직 처리 |
| 예시 | Apache, Nginx | Tomcat, WebLogic, JEUS |
| 처리 방식 | 파일을 그대로 응답 | 프로그램 실행 후 결과 응답 |

전체 흐름은 다음과 같다.

```text
클라이언트
→ 웹 서버
→ WAS
→ 애플리케이션 로직 처리
→ 데이터베이스 접근
→ 응답 반환
```

웹 서버와 WAS를 분리하면 정적 리소스 처리와 동적 로직 처리를 역할별로 나눌 수 있다.

웹 서버는 정적 파일 제공에 집중하고, WAS는 애플리케이션의 핵심 로직 처리에 집중한다.

## 9. Spring과 Spring Boot

Spring과 Spring Boot는 서버 자체라기보다는 백엔드 애플리케이션을 개발하기 위한 프레임워크이다.

Spring 기반 애플리케이션은 보통 다음과 같은 계층 구조를 가진다.

| 계층 | 역할 |
|---|---|
| Controller | 클라이언트의 요청을 받는 계층 |
| Service | 비즈니스 로직을 처리하는 계층 |
| Repository | 데이터베이스 접근을 담당하는 계층 |

요청 흐름은 다음과 같다.

```text
Client Request
→ Controller
→ Service
→ Repository
→ Database
```

Controller는 클라이언트의 요청을 받고, Service는 핵심 비즈니스 로직을 처리하며, Repository는 데이터베이스와의 통신을 담당한다.

이렇게 계층을 나누면 각 코드의 역할이 명확해지고 유지보수가 쉬워진다.

## 10. 데이터베이스와 DBMS

데이터베이스는 데이터를 저장하는 공간이다.  
DBMS는 데이터베이스를 관리하기 위한 시스템이다.

| 용어 | 설명 |
|---|---|
| DB | 데이터를 저장하는 공간 |
| DBMS | DB를 관리하는 소프트웨어 시스템 |

대표적인 DBMS는 다음과 같다.

- Oracle
- MySQL
- PostgreSQL
- Microsoft SQL Server

일반적으로 웹 서비스에서는 사용자 정보, 게시글 정보, 주문 정보, 상품 정보와 같은 데이터를 데이터베이스에 저장한다.

백엔드 애플리케이션은 필요한 경우 데이터베이스에 접근하여 데이터를 조회하거나 저장한다.

## 11. RDBMS와 NoSQL

데이터베이스는 크게 관계형 데이터베이스와 NoSQL 데이터베이스로 나눌 수 있다.

### RDBMS

RDBMS는 Relational Database Management System의 약자이다.  
관계형 데이터베이스 관리 시스템을 의미한다.

RDBMS는 데이터를 테이블 형태로 저장하고, SQL을 사용해 데이터를 다룬다.

대표적인 RDBMS는 다음과 같다.

- Oracle
- MySQL
- PostgreSQL
- Microsoft SQL Server

SQL은 Structured Query Language의 약자이다.  
관계형 데이터베이스에서 데이터를 조회, 삽입, 수정, 삭제하기 위해 사용하는 언어이다.

예시는 다음과 같다.

```sql
SELECT * FROM users;
```

### NoSQL

NoSQL은 관계형 구조가 아닌 방식으로 데이터를 저장하는 데이터베이스이다.

대표적인 NoSQL은 다음과 같다.

- MongoDB
- Redis
- Cassandra

RDBMS와 NoSQL은 데이터를 저장하는 방식과 사용 목적이 다르다.  
RDBMS는 정형화된 테이블 구조에 적합하고, NoSQL은 유연한 데이터 구조나 빠른 처리에 적합한 경우가 많다.

## 12. JDBC

JDBC는 Java Database Connectivity의 약자이다.  
Java 프로그램이 데이터베이스와 연결하여 데이터를 주고받을 수 있도록 도와주는 API이다.

Java 백엔드 애플리케이션은 JDBC를 통해 데이터베이스에 접근할 수 있다.

기본 흐름은 다음과 같다.

```text
Java Application
→ JDBC
→ Database
```

하지만 JDBC를 직접 사용하면 데이터베이스 연결, SQL 실행, 결과 처리 등 반복적인 코드가 많아질 수 있다.  
이러한 불편함을 줄이기 위해 ORM과 같은 기술이 등장했다.

## 13. ORM과 JPA

ORM은 Object Relational Mapping의 약자이다.  
객체와 관계형 데이터베이스의 데이터를 매핑해주는 기술이다.

Java에서는 ORM을 사용하기 위한 표준 API로 JPA를 사용할 수 있다.

| 용어 | 설명 |
|---|---|
| ORM | 객체와 관계형 데이터베이스를 매핑하는 기술 |
| JPA | Java에서 ORM을 사용하기 위한 표준 API |
| MyBatis | SQL 중심으로 DB 접근을 도와주는 프레임워크 |

ORM을 사용하면 SQL을 직접 많이 작성하지 않고도 객체 중심으로 데이터를 다룰 수 있다.

예를 들어 Java에서 `User` 객체를 만들고, 이 객체를 데이터베이스의 `users` 테이블과 연결할 수 있다.

정리하면 다음과 같다.

```text
Java 객체
↔ ORM / JPA
↔ 데이터베이스 테이블
```

## 14. 브라우저 렌더링

서버가 응답을 반환하면 브라우저는 응답으로 받은 리소스를 화면에 표시한다.  
이 과정을 렌더링이라고 한다.

브라우저는 HTML, CSS, JavaScript, 이미지 등의 리소스를 해석하여 사용자가 볼 수 있는 화면을 만든다.

흐름은 다음과 같다.

```text
서버 응답 수신
→ 리소스 해석
→ 화면 구성
→ 사용자에게 표시
```

즉, 서버가 데이터를 응답하면 브라우저는 그 데이터를 기반으로 화면을 구성하고 사용자에게 보여준다.

## 15. 전체 흐름 정리

클라이언트와 서버 기반의 웹 서비스 흐름은 다음과 같이 정리할 수 있다.

```text
사용자
→ 브라우저
→ URL 입력 또는 이벤트 발생
→ HTTP/HTTPS Request
→ 웹 서버
→ WAS
→ Controller
→ Service
→ Repository
→ Database
→ Repository
→ Service
→ Controller
→ Response
→ 브라우저 렌더링
→ 사용자 화면 표시
```

이 흐름에서 각 구성 요소의 역할은 다음과 같다.

| 구성 요소 | 역할 |
|---|---|
| 브라우저 | 사용자가 웹 서비스에 접근하는 도구 |
| URL | 웹 서비스에 접근하기 위한 주소 |
| HTTP/HTTPS | 클라이언트와 서버가 통신하기 위한 프로토콜 |
| 웹 서버 | 정적 리소스 처리 |
| WAS | 동적 리소스와 애플리케이션 로직 처리 |
| Controller | 클라이언트 요청 수신 |
| Service | 비즈니스 로직 처리 |
| Repository | 데이터베이스 접근 |
| Database | 데이터 저장 및 관리 |
| Response | 서버가 클라이언트에게 반환하는 응답 |
| 렌더링 | 브라우저가 응답 결과를 화면에 표시하는 과정 |

## 16. 정리

클라이언트와 서버는 요청과 응답을 주고받으며 웹 서비스를 구성한다.

클라이언트는 브라우저를 통해 서버에 요청을 보내고, 서버는 요청을 처리한 뒤 응답을 반환한다.  
사용자는 URL을 통해 웹 서비스에 접근하고, 클라이언트와 서버는 HTTP 또는 HTTPS 프로토콜을 사용해 통신한다.

서버 측에서는 웹 서버와 WAS가 역할을 나누어 동작할 수 있다.  
웹 서버는 정적 리소스를 처리하고, WAS는 동적 리소스와 애플리케이션 로직을 처리한다.

백엔드 애플리케이션은 Controller, Service, Repository 계층으로 구성될 수 있으며, 필요한 경우 데이터베이스와 통신한다.  
Java에서는 JDBC를 통해 데이터베이스와 연결할 수 있고, ORM과 JPA를 사용하면 객체 중심으로 데이터베이스를 다룰 수 있다.

이번 학습에서 중요한 개념은 다음과 같다.

- 클라이언트는 요청을 보내는 쪽이다.
- 서버는 요청을 처리하고 응답을 반환하는 쪽이다.
- URL은 웹 서비스에 접근하기 위한 주소이다.
- HTTP와 HTTPS는 웹 통신을 위한 프로토콜이다.
- 웹 서버는 정적 리소스를 처리한다.
- WAS는 동적 리소스를 처리한다.
- Spring 애플리케이션은 Controller, Service, Repository 계층으로 구성된다.
- 데이터베이스는 데이터를 저장하고, DBMS는 데이터베이스를 관리한다.
- Java에서는 JDBC, ORM, JPA 등을 통해 데이터베이스와 통신할 수 있다.
