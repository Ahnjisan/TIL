# 웹 서비스 기본 개념

## 1. 웹 서비스란?

웹 서비스는 사용자가 브라우저를 통해 서버에 요청을 보내고, 서버가 요청을 처리한 뒤 다시 응답을 반환하는 구조로 동작하는 서비스이다.

웹 서비스는 단순히 사용자가 보는 화면만 의미하지 않는다.  
사용자의 요청이 프론트엔드, 백엔드, 데이터베이스를 거쳐 처리되고 다시 화면으로 반환되는 전체 흐름을 포함한다.

즉, 웹 서비스는 다음과 같은 요소들이 함께 동작하는 구조이다.

```text
사용자
→ 브라우저
→ 프론트엔드
→ 백엔드
→ 데이터베이스
→ 백엔드
→ 프론트엔드
→ 브라우저
→ 사용자
```

## 2. 웹 서비스의 주요 구성 요소

웹 서비스는 크게 다음과 같은 구성 요소로 이루어진다.

| 구성 요소 | 설명 |
|---|---|
| 클라이언트 | 서비스를 사용하는 사용자 또는 사용자의 브라우저 |
| 프론트엔드 | 사용자가 직접 보는 화면과 UI를 담당하는 영역 |
| 백엔드 | 사용자의 요청을 처리하고 비즈니스 로직을 수행하는 영역 |
| 데이터베이스 | 서비스에 필요한 데이터를 저장하고 관리하는 영역 |

각 구성 요소는 역할이 분리되어 있지만, 실제 서비스에서는 요청과 응답을 중심으로 서로 연결되어 동작한다.

## 3. 클라이언트

클라이언트는 서비스를 요청하는 주체이다.  
웹 서비스에서는 일반적으로 사용자가 사용하는 브라우저를 클라이언트라고 볼 수 있다.

사용자는 브라우저를 통해 다음과 같은 동작을 수행한다.

- 버튼 클릭
- 검색어 입력
- 로그인 요청
- 게시글 조회
- 데이터 입력

이러한 동작은 이벤트가 되고, 이벤트를 통해 서버에 요청이 전달된다.

예를 들어 사용자가 로그인 버튼을 클릭하면 브라우저는 서버에 로그인 요청을 보낸다.  
서버는 요청을 처리한 뒤 로그인 성공 또는 실패 결과를 다시 브라우저에 응답한다.

## 4. 프론트엔드

프론트엔드는 사용자가 직접 보는 화면 영역이다.  
사용자는 백엔드나 데이터베이스에 직접 접근하지 않고, 프론트엔드를 통해 서비스와 상호작용한다.

프론트엔드의 주요 역할은 다음과 같다.

- 사용자 화면 구성
- UI 제공
- 사용자 이벤트 처리
- 백엔드 API 호출
- 서버 응답 데이터를 화면에 표시
- 사용자 경험 개선

프론트엔드는 사용자의 입력을 받아 백엔드에 요청을 보내고, 백엔드로부터 받은 응답을 화면에 보여준다.

예를 들어 게시글 목록 화면에서 사용자가 게시글을 조회하면, 프론트엔드는 백엔드에 게시글 데이터를 요청하고, 응답받은 데이터를 화면에 렌더링한다.

## 5. UI와 UX

UI는 User Interface의 약자이다.  
사용자가 서비스를 이용할 때 직접 마주하는 화면이나 조작 요소를 의미한다.

UI의 예시는 다음과 같다.

- 버튼
- 입력창
- 메뉴
- 검색창
- 화면 레이아웃
- 아이콘

UX는 User Experience의 약자이다.  
사용자가 서비스를 이용하면서 느끼는 전체적인 경험을 의미한다.

프론트엔드는 단순히 화면을 예쁘게 만드는 것뿐만 아니라, 사용자가 서비스를 편리하고 직관적으로 사용할 수 있도록 UI와 UX를 함께 고려해야 한다.

예를 들어 로그인 버튼의 위치가 명확하고, 오류 메시지가 이해하기 쉽게 표시된다면 사용자 경험이 좋아질 수 있다.

## 6. 백엔드

백엔드는 사용자의 요청을 실제로 처리하는 영역이다.  
프론트엔드에서 전달된 요청을 받아 비즈니스 로직을 수행하고, 필요한 경우 데이터베이스와 통신한다.

백엔드의 주요 역할은 다음과 같다.

- 클라이언트 요청 처리
- 비즈니스 로직 수행
- 데이터 검증
- 데이터베이스 연동
- 인증 및 권한 처리
- API 응답 반환

프론트엔드가 사용자가 보는 화면을 담당한다면, 백엔드는 서비스의 핵심 기능과 데이터 처리를 담당한다.

예를 들어 로그인 기능에서 백엔드는 사용자가 입력한 아이디와 비밀번호가 올바른지 검증하고, 데이터베이스에 저장된 사용자 정보와 비교한 뒤 결과를 반환한다.

## 7. 데이터베이스

데이터베이스는 서비스에 필요한 데이터를 저장하고 관리하는 공간이다.

웹 서비스에서는 다음과 같은 데이터가 데이터베이스에 저장될 수 있다.

- 회원 정보
- 게시글 정보
- 댓글 정보
- 주문 정보
- 상품 정보
- 로그인 기록

사용자는 데이터베이스에 직접 접근하지 않는다.  
사용자는 프론트엔드를 통해 요청을 보내고, 백엔드는 데이터베이스에 접근하여 필요한 데이터를 조회하거나 저장한다.

예를 들어 사용자가 게시글 목록을 조회하면 백엔드는 데이터베이스에서 게시글 데이터를 가져오고, 그 결과를 프론트엔드에 응답으로 반환한다.

## 8. Request와 Response

웹 서비스의 핵심 흐름은 Request와 Response이다.

| 용어 | 의미 |
|---|---|
| Request | 클라이언트가 서버에 보내는 요청 |
| Response | 서버가 클라이언트에게 반환하는 응답 |

웹 서비스는 클라이언트가 서버에 요청을 보내고, 서버가 요청을 처리한 뒤 응답을 반환하는 방식으로 동작한다.

예를 들어 로그인 요청의 흐름은 다음과 같다.

```text
사용자 로그인 버튼 클릭
→ 프론트엔드에서 로그인 요청 생성
→ 백엔드 서버로 Request 전송
→ 백엔드에서 로그인 정보 검증
→ 데이터베이스에서 사용자 정보 확인
→ 로그인 결과 Response 반환
→ 프론트엔드에서 결과 표시
```

즉, 웹 서비스는 요청과 응답이 반복되면서 동작하는 구조이다.

## 9. 프론트엔드와 백엔드의 차이

프론트엔드와 백엔드는 담당하는 역할이 다르다.

| 구분 | 프론트엔드 | 백엔드 |
|---|---|---|
| 담당 영역 | 사용자가 보는 화면 | 서버 내부 로직 |
| 주요 역할 | UI 구성, 이벤트 처리, 화면 렌더링 | 요청 처리, 비즈니스 로직, DB 연동 |
| 사용자와의 관계 | 사용자가 직접 접함 | 사용자가 직접 접하지 않음 |
| 주요 관심사 | UI, UX, 화면 흐름 | 데이터 처리, 로직, 보안, 성능 |
| 예시 기술 | HTML, CSS, JavaScript, React | Java, Spring, Spring Boot, DB |

프론트엔드는 데이터를 화면에 보여주는 역할에 집중하고, 백엔드는 데이터를 처리하고 관리하는 역할에 집중한다.

예를 들어 사용자가 상품 목록 페이지에 접속하면 프론트엔드는 상품 데이터를 화면에 보기 좋게 보여준다.  
하지만 어떤 상품 데이터를 가져올지, 데이터베이스에서 어떻게 조회할지, 가격이나 재고 정보를 어떻게 처리할지는 백엔드에서 담당한다.

## 10. 개발자의 역할 구분

웹 서비스 개발에서는 담당 영역에 따라 개발자의 역할이 나뉜다.

| 역할 | 설명 |
|---|---|
| 프론트엔드 개발자 | 사용자가 보는 화면과 UI를 개발 |
| 백엔드 개발자 | 서버 로직과 API, 데이터 처리를 개발 |
| DBA | 데이터베이스 설계, 관리, 운영 담당 |
| DevOps | 개발된 서비스를 배포하고 운영 환경을 관리 |
| 풀스택 개발자 | 프론트엔드, 백엔드, 데이터베이스 등 여러 영역을 함께 다루는 개발자 |

각 역할은 독립적으로 존재하는 것이 아니라, 하나의 서비스를 만들기 위해 서로 협업한다.

예를 들어 프론트엔드 개발자는 화면을 만들고, 백엔드 개발자는 API를 만들며, DBA는 데이터베이스 구조를 관리하고, DevOps는 서비스가 안정적으로 배포되고 운영될 수 있도록 환경을 구성한다.

## 11. DevOps

DevOps는 Development와 Operations의 합성어이다.  
개발과 운영을 함께 고려하는 문화 또는 역할을 의미한다.

전통적으로 개발자는 코드를 작성하고, 운영자는 서버 배포와 운영을 담당했다.  
하지만 최근에는 개발자가 배포와 운영 환경까지 이해해야 하는 경우가 많아지면서 DevOps의 중요성이 커졌다.

DevOps에서 다루는 영역은 다음과 같다.

- 배포 자동화
- 서버 운영
- CI/CD
- 모니터링
- 인프라 관리
- 장애 대응

웹 서비스는 개발만으로 끝나는 것이 아니라 사용자에게 안정적으로 제공되어야 한다.  
따라서 서비스를 배포하고 운영하는 과정도 웹 서비스 개발에서 중요한 영역이다.

## 12. API

API는 Application Programming Interface의 약자이다.  
웹 서비스에서는 프론트엔드와 백엔드가 서로 통신하기 위한 규칙이라고 이해할 수 있다.

프론트엔드와 백엔드는 API를 통해 다음과 같은 내용을 약속한다.

- 어떤 URL로 요청할 것인지
- 어떤 데이터를 요청에 담을 것인지
- 어떤 방식으로 요청할 것인지
- 서버는 어떤 형식으로 응답할 것인지
- 오류가 발생하면 어떤 응답을 줄 것인지

예를 들어 로그인 API는 다음과 같이 정의할 수 있다.

```text
POST /api/login
```

요청 데이터는 다음과 같을 수 있다.

```json
{
  "email": "user@example.com",
  "password": "1234"
}
```

응답 데이터는 다음과 같을 수 있다.

```json
{
  "success": true,
  "message": "로그인 성공"
}
```

API는 프론트엔드와 백엔드가 협업하기 위한 약속이다.  
API 규칙이 명확하지 않으면 프론트엔드와 백엔드를 각각 개발한 뒤 통합하는 과정에서 오류가 발생할 수 있다.

## 13. API 명세

API 명세는 API의 사용 방법을 문서로 정리한 것이다.

API 명세에는 보통 다음과 같은 내용이 포함된다.

- API 이름
- 요청 URL
- HTTP Method
- 요청 데이터 형식
- 응답 데이터 형식
- 에러 응답 형식

예를 들어 회원가입 API 명세는 다음과 같이 작성할 수 있다.

| 항목 | 내용 |
|---|---|
| API 이름 | 회원가입 |
| URL | `/api/users/signup` |
| Method | POST |
| Request Body | email, password, name |
| Response | 회원가입 성공 여부 |

API 명세를 미리 작성하면 프론트엔드와 백엔드가 같은 기준으로 개발할 수 있다.  
또한 협업 과정에서 요청과 응답 형식이 맞지 않아 발생하는 문제를 줄일 수 있다.

## 14. Swagger

Swagger는 API 명세를 문서화하고 테스트할 수 있도록 도와주는 도구이다.

Swagger를 사용하면 다음과 같은 내용을 확인할 수 있다.

- API 목록
- 요청 URL
- HTTP Method
- 요청 파라미터
- 요청 Body 형식
- 응답 형식
- 에러 응답

Swagger는 프론트엔드 개발자와 백엔드 개발자가 API를 기준으로 협업할 때 유용하다.

예를 들어 백엔드 개발자가 Swagger에 API 명세를 정리해두면, 프론트엔드 개발자는 해당 문서를 보고 어떤 URL로 어떤 데이터를 보내야 하는지 확인할 수 있다.

## 15. 웹 서비스 전체 흐름 예시

사용자가 게시글 목록을 조회하는 상황을 예로 들면 다음과 같다.

```text
사용자
→ 브라우저에서 게시글 목록 페이지 접속
→ 프론트엔드가 게시글 목록 API 호출
→ 백엔드가 요청 수신
→ 백엔드가 데이터베이스에서 게시글 목록 조회
→ 백엔드가 조회 결과를 Response로 반환
→ 프론트엔드가 응답 데이터를 화면에 렌더링
→ 사용자가 게시글 목록 확인
```

이 과정에서 각 구성 요소는 다음과 같은 역할을 한다.

| 구성 요소 | 역할 |
|---|---|
| 사용자 | 게시글 목록 조회 요청 |
| 브라우저 | 웹 서비스에 접근하는 도구 |
| 프론트엔드 | API 호출 및 화면 표시 |
| 백엔드 | 요청 처리 및 데이터 조회 |
| 데이터베이스 | 게시글 데이터 저장 |
| API | 프론트엔드와 백엔드 사이의 통신 규칙 |

## 16. 정리

웹 서비스는 사용자의 요청과 서버의 응답을 중심으로 동작한다.

사용자는 브라우저를 통해 프론트엔드와 상호작용하고, 프론트엔드는 사용자의 요청을 백엔드로 전달한다.  
백엔드는 비즈니스 로직을 처리하고 필요한 경우 데이터베이스와 통신한 뒤 결과를 반환한다.  
프론트엔드는 반환받은 결과를 화면에 표시하고, 사용자는 브라우저를 통해 결과를 확인한다.

핵심 흐름은 다음과 같다.

```text
클라이언트 요청
→ 프론트엔드
→ 백엔드
→ 데이터베이스
→ 백엔드
→ 프론트엔드
→ 클라이언트 응답
```

이번 학습에서 중요한 개념은 다음과 같다.

- 웹 서비스는 Request와 Response를 중심으로 동작한다.
- 클라이언트는 서비스를 요청하는 쪽이다.
- 프론트엔드는 사용자가 직접 보는 화면을 담당한다.
- 백엔드는 사용자의 요청을 처리하고 비즈니스 로직을 수행한다.
- 데이터베이스는 서비스에 필요한 데이터를 저장한다.
- API는 프론트엔드와 백엔드가 통신하기 위한 규칙이다.
- API 명세는 협업 과정에서 요청과 응답 형식을 맞추기 위해 필요하다.
- Swagger는 API 명세를 문서화하고 확인할 수 있도록 도와주는 도구이다.
