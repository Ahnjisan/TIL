# Merge, Rebase, Conflict 이해하기

## 1. 브랜치를 합치는 이유

Git에서 브랜치는 독립적인 작업 공간이다.

기능 개발, 테스트, 실험 작업 등을 각각 다른 브랜치에서 진행하면 main 브랜치를 안정적으로 유지할 수 있다.

하지만 브랜치에서 작업한 내용은 최종적으로 기준 브랜치에 합쳐야 한다.

예를 들어 `add-coach` 브랜치에서 코치 정보를 추가하고, `new-teams` 브랜치에서 새로운 팀 정보를 추가했다면 이 작업들은 최종적으로 `main` 브랜치에 반영되어야 한다.

브랜치를 합치는 대표적인 방법은 다음과 같다.

| 방식 | 설명 |
|---|---|
| Merge | 두 브랜치를 하나의 병합 커밋으로 합침 |
| Rebase | 한 브랜치의 커밋을 다른 브랜치 뒤에 이어 붙임 |

## 2. Merge란?

Merge는 두 브랜치를 하나로 합치는 방식이다.

기준이 되는 브랜치로 이동한 뒤, 합치고 싶은 브랜치를 merge 한다.

예를 들어 `main` 브랜치에 `add-coach` 브랜치를 병합하려면 다음과 같이 진행한다.

    git switch main
    git merge add-coach

이 명령어는 `main` 브랜치 입장에서 `add-coach` 브랜치의 작업 내용을 가져오는 것이다.

Merge를 하면 브랜치의 작업 흐름이 이력에 남는다.

따라서 어떤 브랜치에서 어떤 작업이 진행되었고, 언제 main에 병합되었는지 확인하기 쉽다.

## 3. Merge의 특징

Merge의 특징은 다음과 같다.

| 특징 | 설명 |
|---|---|
| 이력 보존 | 브랜치가 분기되고 병합된 흐름이 남음 |
| 병합 커밋 | 상황에 따라 merge commit이 생성됨 |
| 협업 적합성 | 여러 사람이 작업한 흐름을 추적하기 좋음 |
| 단점 | 커밋 그래프가 복잡해질 수 있음 |

Merge는 브랜치 작업 흐름을 보존하고 싶을 때 적합하다.

특히 협업 프로젝트에서는 어떤 브랜치에서 어떤 작업이 진행되었는지 남기는 것이 중요하기 때문에 Merge 방식이 많이 사용된다.

## 4. Merge 실습 흐름

강의에서는 `main` 브랜치에 `add-coach` 브랜치를 병합하는 실습을 진행했다.

전체 흐름은 다음과 같다.

    main 브랜치로 이동
    → add-coach 브랜치 병합
    → 병합 커밋 생성
    → SourceTree에서 병합 이력 확인
    → 병합 완료된 브랜치 삭제

명령어는 다음과 같다.

    git switch main
    git merge add-coach
    git branch -d add-coach

`add-coach` 브랜치에서 작업한 내용이 `main` 브랜치에 반영되면, 더 이상 `add-coach` 브랜치를 유지할 필요가 없을 수 있다.

이때 `git branch -d` 명령어로 병합이 끝난 브랜치를 삭제할 수 있다.

## 5. 병합 후 브랜치 삭제

브랜치를 삭제할 때는 다음 명령어를 사용한다.

    git branch -d 브랜치명

예시는 다음과 같다.

    git branch -d add-coach

브랜치를 삭제해도 이미 `main` 브랜치에 병합된 내용은 사라지지 않는다.

삭제되는 것은 브랜치 이름일 뿐이고, 병합된 커밋 내용은 `main` 브랜치에 남아 있다.

정리하면 다음과 같다.

    add-coach 브랜치 작업 완료
    → main 브랜치로 merge
    → add-coach 브랜치 삭제
    → main에는 작업 내용 유지

## 6. Rebase란?

Rebase는 한 브랜치의 커밋을 다른 브랜치 뒤에 이어 붙이는 방식이다.

Merge가 브랜치의 흐름을 남기면서 합치는 방식이라면, Rebase는 커밋 이력을 한 줄로 정리하는 방식이다.

예를 들어 `new-teams` 브랜치의 커밋을 `main` 브랜치 뒤에 이어 붙이려면 다음과 같이 진행한다.

    git switch new-teams
    git rebase main

이 명령어는 `new-teams` 브랜치의 커밋들을 `main` 브랜치의 최신 커밋 뒤로 옮겨 붙인다.

## 7. Rebase의 특징

Rebase의 특징은 다음과 같다.

| 특징 | 설명 |
|---|---|
| 이력 정리 | 커밋 이력이 한 줄처럼 정리됨 |
| 병합 커밋 | 일반적으로 별도 병합 커밋이 생기지 않음 |
| 장점 | 커밋 로그가 깔끔해짐 |
| 주의점 | 이미 공유된 커밋에는 신중하게 사용해야 함 |

Rebase는 커밋 이력을 깔끔하게 유지하고 싶을 때 적합하다.

하지만 이미 팀원과 공유된 커밋에 Rebase를 사용하면 커밋 이력이 변경되어 협업에 문제가 생길 수 있다.

따라서 협업 중인 브랜치에서는 Rebase 사용에 주의해야 한다.

## 8. Rebase 실습 흐름

강의에서는 `new-teams` 브랜치를 `main` 브랜치 뒤에 이어 붙이는 실습을 진행했다.

전체 흐름은 다음과 같다.

    new-teams 브랜치로 이동
    → main 브랜치를 기준으로 rebase
    → main 브랜치로 이동
    → fast-forward merge 수행
    → 병합 완료된 브랜치 삭제

명령어는 다음과 같다.

    git switch new-teams
    git rebase main
    git switch main
    git merge new-teams
    git branch -d new-teams

Rebase 이후 `main` 브랜치를 최신 커밋 위치로 이동시키기 위해 `git merge new-teams`를 수행한다.

이때 새로운 병합 커밋이 생기지 않고 main 브랜치가 앞으로 이동하는 경우를 Fast-forward라고 한다.

## 9. Fast-forward란?

Fast-forward는 브랜치 포인터가 최신 커밋 위치로 단순히 앞으로 이동하는 방식이다.

예를 들어 `main` 브랜치가 `new-teams` 브랜치보다 뒤에 있고, 충돌 없이 그대로 따라갈 수 있는 상황이라면 Git은 새로운 병합 커밋을 만들지 않고 `main` 브랜치 위치만 앞으로 이동시킨다.

흐름은 다음과 같다.

    main 브랜치가 뒤에 있음
    → new-teams 브랜치의 최신 커밋까지 이동 가능
    → git merge new-teams
    → main 브랜치 포인터가 앞으로 이동

Fast-forward는 커밋 이력을 깔끔하게 유지할 수 있다는 장점이 있다.

## 10. Merge와 Rebase의 차이

Merge와 Rebase는 모두 브랜치를 합치는 방법이지만 이력 관리 방식이 다르다.

| 구분 | Merge | Rebase |
|---|---|---|
| 개념 | 두 브랜치를 병합 커밋으로 합침 | 커밋을 다른 브랜치 뒤에 이어 붙임 |
| 이력 형태 | 브랜치 분기와 병합 흐름이 남음 | 한 줄로 정리됨 |
| 병합 커밋 | 생성될 수 있음 | 일반적으로 생성되지 않음 |
| 장점 | 작업 흐름 추적이 쉬움 | 커밋 이력이 깔끔함 |
| 주의점 | 이력이 복잡해질 수 있음 | 공유된 커밋에 사용하면 위험할 수 있음 |

정리하면 Merge는 작업 흐름을 남기는 방식이고, Rebase는 이력을 정리하는 방식이다.

## 11. Conflict란?

Conflict는 서로 다른 브랜치에서 같은 파일의 같은 부분을 수정했을 때 Git이 자동으로 병합하지 못하는 상황이다.

예를 들어 `main` 브랜치와 `feature` 브랜치에서 같은 파일의 같은 줄을 서로 다르게 수정했다고 가정할 수 있다.

이 경우 Git은 어떤 내용을 최종 결과로 선택해야 할지 알 수 없기 때문에 Conflict를 발생시킨다.

Conflict는 보통 다음 상황에서 발생한다.

- 같은 파일의 같은 줄을 서로 다르게 수정한 경우
- 한 브랜치에서는 파일을 수정하고 다른 브랜치에서는 삭제한 경우
- 여러 브랜치에서 같은 설정 파일을 다르게 수정한 경우

## 12. Conflict가 발생했을 때의 표시

Conflict가 발생하면 파일 안에 다음과 같은 표시가 생긴다.

    <<<<<<< HEAD
    현재 브랜치의 내용
    =======
    병합하려는 브랜치의 내용
    >>>>>>> feature-branch

각 구분의 의미는 다음과 같다.

| 표시 | 의미 |
|---|---|
| `<<<<<<< HEAD` | 현재 브랜치의 변경 내용 시작 |
| `=======` | 두 변경 내용의 구분선 |
| `>>>>>>> 브랜치명` | 병합하려는 브랜치의 변경 내용 끝 |

개발자는 이 표시를 보고 어떤 내용을 남길지 직접 결정해야 한다.

## 13. Conflict 해결 흐름

Conflict가 발생했을 때는 다음 순서로 해결한다.

    Conflict 발생
    → 충돌 파일 확인
    → 충돌 표시 확인
    → 남길 내용 선택
    → 충돌 표시 제거
    → 파일 저장
    → git add
    → git commit

명령어 흐름은 다음과 같다.

    git status
    충돌 파일 수정
    git add .
    git commit -m "resolve merge conflict"

Conflict 해결 후에는 반드시 `git add`를 실행해야 한다.

Git은 충돌이 해결된 파일을 Staging Area에 올린 뒤 커밋해야 병합이 완료되었다고 판단한다.

## 14. Conflict 해결 시 주의할 점

Conflict를 해결할 때는 다음 사항을 주의해야 한다.

- 충돌 표시를 반드시 제거해야 한다.
- 어떤 내용을 남길지 기능 기준으로 판단해야 한다.
- 상대 브랜치의 변경 내용을 무조건 삭제하면 안 된다.
- 충돌 해결 후 테스트를 실행해야 한다.
- 해결한 내용을 커밋으로 기록해야 한다.

Conflict는 오류라기보다는 Git이 자동으로 판단할 수 없는 부분을 개발자에게 맡기는 상황이다.

따라서 충돌 내용을 직접 확인하고 올바른 최종 상태로 수정하는 것이 중요하다.

## 15. 정리

이번 학습에서는 브랜치를 병합하는 방법인 Merge와 Rebase를 학습했다.

Merge는 두 브랜치를 병합 커밋으로 합치는 방식이고, Rebase는 한 브랜치의 커밋을 다른 브랜치 뒤에 이어 붙이는 방식이다.

또한 브랜치를 합치는 과정에서 발생할 수 있는 Conflict의 개념도 정리했다.

핵심 개념은 다음과 같다.

- 브랜치에서 작업한 내용은 최종적으로 기준 브랜치에 병합해야 한다.
- Merge는 브랜치의 작업 흐름을 남기면서 병합하는 방식이다.
- Rebase는 커밋 이력을 한 줄로 정리하는 방식이다.
- Fast-forward는 브랜치 포인터가 최신 커밋으로 이동하는 방식이다.
- 병합이 완료된 브랜치는 삭제할 수 있다.
- Conflict는 Git이 자동으로 병합하지 못하는 충돌 상황이다.
- Conflict가 발생하면 개발자가 직접 최종 내용을 선택해야 한다.
- Conflict 해결 후에는 `git add`와 `git commit`을 통해 병합을 완료해야 한다.
