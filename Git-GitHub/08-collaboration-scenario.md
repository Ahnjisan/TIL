# GitHub 협업 시나리오 정리

## 1. 협업에서 원격 저장소가 필요한 이유

혼자 개발할 때는 로컬 저장소만으로도 Git을 사용할 수 있다.

하지만 여러 사람이 함께 개발하는 프로젝트에서는 코드를 공유할 수 있는 공간이 필요하다.

이때 사용하는 것이 GitHub 같은 원격 저장소이다.

원격 저장소를 사용하면 다음과 같은 작업이 가능하다.

- 팀원과 코드 공유
- 프로젝트 이력 관리
- 브랜치 기반 기능 개발
- Pull Request 기반 코드 리뷰
- Issue 기반 작업 관리
- GitHub Actions 기반 자동화

즉, GitHub는 Git으로 관리되는 프로젝트를 온라인에서 공유하고 협업할 수 있도록 도와주는 플랫폼이다.

## 2. 기본 협업 흐름

GitHub를 사용하는 기본 협업 흐름은 다음과 같다.

    원격 저장소 생성
    → 팀원이 clone
    → 작업 전 pull
    → 새 브랜치 생성
    → 기능 개발
    → add, commit
    → push
    → Pull Request 생성
    → 코드 리뷰
    → main 브랜치에 병합

협업에서는 main 브랜치에 직접 작업하기보다 기능별 브랜치를 만들어 작업하는 것이 일반적이다.

이렇게 하면 main 브랜치를 안정적인 상태로 유지할 수 있다.

## 3. Clone으로 프로젝트 가져오기

팀원이 GitHub에 있는 프로젝트를 처음 가져올 때는 clone을 사용한다.

    git clone 원격저장소URL

Clone을 하면 프로젝트 파일뿐만 아니라 Git 이력까지 함께 내려받는다.

Clone 이후에는 해당 프로젝트 폴더 안에서 Git 명령어를 사용할 수 있다.

예시는 다음과 같다.

    git clone https://github.com/username/project.git
    cd project

Clone은 협업 프로젝트에 처음 참여할 때 가장 먼저 수행하는 작업이다.

## 4. 작업 전 최신 상태 맞추기

협업에서는 작업을 시작하기 전에 원격 저장소의 최신 변경 사항을 가져와야 한다.

    git pull

또는 명시적으로 다음과 같이 작성할 수 있다.

    git pull origin main

작업 전 pull을 하지 않으면 다른 팀원이 이미 수정한 내용을 모른 채 작업할 수 있다.

이 경우 나중에 push가 거부되거나 conflict가 발생할 수 있다.

따라서 협업에서는 작업 시작 전에 항상 최신 상태를 확인하는 습관이 중요하다.

## 5. 기능 브랜치 생성

main 브랜치에서 직접 작업하면 안정적인 코드가 깨질 수 있다.

따라서 새로운 기능을 개발할 때는 별도 브랜치를 생성하는 것이 좋다.

브랜치 생성과 이동을 동시에 하려면 다음 명령어를 사용한다.

    git switch -c feature/login

브랜치 이름은 작업 내용을 알 수 있도록 작성하는 것이 좋다.

예시는 다음과 같다.

| 브랜치 이름 | 의미 |
|---|---|
| feature/login | 로그인 기능 개발 |
| feature/signup | 회원가입 기능 개발 |
| fix/login-error | 로그인 오류 수정 |
| docs/readme | README 수정 |

## 6. 로컬에서 작업하고 커밋하기

기능 브랜치를 만든 뒤에는 파일을 수정하고 커밋한다.

기본 흐름은 다음과 같다.

    파일 수정
    → git status
    → git add .
    → git commit -m "커밋 메시지"

예시는 다음과 같다.

    git status
    git add .
    git commit -m "로그인 기능 추가"

커밋 메시지는 작업 내용을 이해할 수 있도록 명확하게 작성해야 한다.

좋지 않은 예시는 다음과 같다.

    수정
    test
    작업함

좋은 예시는 다음과 같다.

    로그인 API 요청 로직 추가
    회원가입 폼 유효성 검사 추가
    README 실행 방법 수정

## 7. 원격 저장소에 브랜치 Push하기

로컬에서 만든 브랜치와 커밋은 아직 내 컴퓨터에만 존재한다.

다른 팀원이 확인하거나 Pull Request를 만들기 위해서는 원격 저장소에 push해야 한다.

    git push origin feature/login

처음 push하는 브랜치라면 다음처럼 upstream을 설정할 수도 있다.

    git push -u origin feature/login

이후부터는 간단히 다음 명령어를 사용할 수 있다.

    git push

Push를 하면 GitHub에서 해당 브랜치를 확인할 수 있다.

## 8. Pull Request 생성

Push한 브랜치의 변경 사항을 main 브랜치에 반영하려면 Pull Request를 생성한다.

Pull Request는 내 브랜치의 변경 사항을 기준 브랜치에 병합해달라고 요청하는 기능이다.

기본 흐름은 다음과 같다.

    기능 브랜치 push
    → GitHub에서 Pull Request 생성
    → 변경 내용 확인
    → 코드 리뷰
    → 수정 요청 반영
    → 승인 후 main에 병합

Pull Request를 사용하면 main 브랜치에 바로 반영하지 않고, 변경 내용을 검토한 뒤 병합할 수 있다.

## 9. GitHub에서 직접 파일을 수정하는 경우

GitHub 웹 화면에서도 파일을 직접 수정할 수 있다.

예를 들어 GitHub에서 `README.md` 파일을 수정하고 commit하면 원격 저장소에 새로운 커밋이 생긴다.

이 경우 로컬 저장소에는 아직 해당 커밋이 없다.

따라서 로컬에서 작업을 계속하려면 먼저 pull을 해야 한다.

    git pull

만약 pull하지 않고 로컬에서 push를 시도하면 원격 저장소에 더 최신 커밋이 있기 때문에 push가 거부될 수 있다.

## 10. 로컬과 원격의 상태가 달라지는 상황

협업에서는 로컬 저장소와 원격 저장소의 상태가 달라지는 일이 자주 발생한다.

예를 들어 다음 상황을 생각할 수 있다.

    로컬에서 파일 수정 후 commit
    → GitHub에서도 같은 저장소에 다른 commit 발생
    → 로컬은 원격의 최신 변경 사항을 모름
    → 바로 push 시도
    → push 거부 가능

이 경우 원격 저장소의 변경 사항을 먼저 가져와야 한다.

    git pull
    git push

즉, push 전에 원격 저장소에 새로운 변경 사항이 있는지 확인하는 것이 중요하다.

## 11. Push가 거부되었을 때 해결 흐름

Push가 거부되면 보통 원격 저장소에 내 로컬에는 없는 커밋이 존재한다는 의미이다.

해결 흐름은 다음과 같다.

    git pull
    충돌이 없다면 자동 병합
    충돌이 있다면 직접 해결
    git add .
    git commit
    git push

만약 pull 과정에서 충돌이 없다면 바로 push하면 된다.

충돌이 발생하면 충돌 파일을 수정하고 다시 add와 commit을 해야 한다.

## 12. Pull 과정에서 발생하는 충돌

Pull은 원격 저장소의 변경 사항을 가져와 현재 로컬 브랜치에 병합하는 작업이다.

따라서 로컬과 원격에서 같은 파일의 같은 부분을 다르게 수정했다면 pull 과정에서도 conflict가 발생할 수 있다.

Conflict가 발생하면 파일 안에 다음과 같은 표시가 생길 수 있다.

    <<<<<<< HEAD
    로컬의 변경 내용
    =======
    원격에서 가져온 변경 내용
    >>>>>>> origin/main

이때 개발자는 어떤 내용을 남길지 직접 결정해야 한다.

해결 흐름은 다음과 같다.

    충돌 파일 확인
    → 남길 내용 선택
    → 충돌 표시 제거
    → 파일 저장
    → git add .
    → git commit
    → git push

## 13. Pull 방식 선택

Pull은 merge 방식 또는 rebase 방식으로 수행될 수 있다.

| 방식 | 설명 |
|---|---|
| Merge 방식 | 원격 변경 사항과 로컬 변경 사항을 병합 커밋으로 합침 |
| Rebase 방식 | 로컬 커밋을 원격 최신 커밋 뒤에 이어 붙임 |

Merge 방식은 다음과 같이 사용할 수 있다.

    git pull --no-rebase

Rebase 방식은 다음과 같이 사용할 수 있다.

    git pull --rebase

팀 프로젝트에서는 어떤 방식을 사용할지 팀 규칙을 정해두는 것이 좋다.

## 14. 협업 시 권장 작업 순서

협업에서 권장되는 작업 순서는 다음과 같다.

    1. main 브랜치로 이동
    2. git pull로 최신 상태 가져오기
    3. 새 기능 브랜치 생성
    4. 기능 개발
    5. git add, git commit
    6. 원격 브랜치로 push
    7. Pull Request 생성
    8. 코드 리뷰
    9. main에 병합
    10. 작업 브랜치 삭제

명령어 예시는 다음과 같다.

    git switch main
    git pull origin main
    git switch -c feature/example
    git add .
    git commit -m "예시 기능 추가"
    git push origin feature/example

이 흐름을 지키면 main 브랜치를 안정적으로 유지하면서 협업할 수 있다.

## 15. main 브랜치에 직접 push하지 않는 이유

협업에서는 main 브랜치에 직접 push하지 않는 것이 좋다.

main 브랜치는 보통 안정적인 코드가 유지되어야 하는 기준 브랜치이다.

main에 직접 push하면 다음 문제가 발생할 수 있다.

- 코드 리뷰 없이 변경 사항이 반영된다.
- 테스트되지 않은 코드가 main에 들어갈 수 있다.
- 다른 팀원의 작업과 충돌할 수 있다.
- 문제가 발생했을 때 원인 추적이 어려워질 수 있다.

따라서 기능 브랜치에서 작업하고 Pull Request를 통해 main에 병합하는 방식이 더 안전하다.

## 16. 협업에서 중요한 습관

GitHub 협업에서 중요한 습관은 다음과 같다.

- 작업 전 항상 pull로 최신 상태 확인
- main 브랜치에서 직접 작업하지 않기
- 기능별로 브랜치 나누기
- 커밋 메시지를 명확하게 작성하기
- push 전 git status 확인하기
- Pull Request로 코드 리뷰 받기
- 충돌 발생 시 내용을 직접 확인하고 해결하기
- 병합 완료 후 사용하지 않는 브랜치 삭제하기

이 습관을 지키면 협업 과정에서 충돌과 혼선을 줄일 수 있다.

## 17. 정리

GitHub 협업은 로컬 저장소와 원격 저장소를 동기화하면서 진행된다.

핵심 개념은 다음과 같다.

- 협업에서는 원격 저장소가 필요하다.
- 프로젝트를 처음 가져올 때는 `git clone`을 사용한다.
- 작업 전에는 `git pull`로 최신 상태를 맞춘다.
- 기능 개발은 별도 브랜치에서 진행하는 것이 좋다.
- 로컬 커밋은 `git push`를 해야 GitHub에 반영된다.
- Pull Request는 변경 사항을 main에 병합해달라고 요청하는 기능이다.
- 원격 저장소에 최신 커밋이 있으면 바로 push가 거부될 수 있다.
- push가 거부되면 먼저 pull을 해야 한다.
- pull 과정에서도 conflict가 발생할 수 있다.
- main 브랜치는 안정적인 상태로 유지하는 것이 중요하다.
