# 버전 관리란?

## 1. 버전 관리가 필요한 이유

개발 프로젝트는 보통 여러 사람이 함께 작업한다.

여러 개발자가 같은 프로젝트를 수정하다 보면 다음과 같은 문제가 발생할 수 있다.

- 누가 어떤 파일을 수정했는지 확인하기 어렵다.
- 어떤 코드가 최신 버전인지 구분하기 어렵다.
- 잘못 수정한 코드를 이전 상태로 되돌리기 어렵다.
- 여러 사람이 동시에 수정한 내용을 병합해야 한다.
- 파일을 복사해서 관리하면 버전이 계속 늘어나 관리가 복잡해진다.

예를 들어 문서를 작성할 때 다음과 같이 파일명을 바꿔가며 저장하는 경우가 있다.

    report.docx
    report_final.docx
    report_final_final.docx
    report_real_final.docx

이 방식은 단순하지만, 파일이 많아질수록 어떤 파일이 최종 버전인지 확인하기 어렵다.

버전 관리는 이러한 문제를 해결하기 위해 필요하다.

버전 관리를 사용하면 파일의 변경 이력을 기록할 수 있고, 특정 시점의 상태를 다시 확인하거나 복구할 수 있다.

## 2. 버전 관리란?

버전 관리는 파일이나 프로젝트의 변경 이력을 기록하고 관리하는 방식이다.

버전 관리를 통해 다음과 같은 작업을 할 수 있다.

| 기능 | 설명 |
|---|---|
| 변경 이력 확인 | 파일이 언제, 어떻게 변경되었는지 확인 |
| 이전 상태 복구 | 문제가 생겼을 때 과거 버전으로 되돌림 |
| 변경 내용 비교 | 현재 상태와 과거 상태의 차이 확인 |
| 협업 지원 | 여러 사람이 수정한 내용을 관리하고 병합 |
| 백업 효과 | 파일 손실이나 실수를 복구할 수 있음 |

즉, 버전 관리는 프로젝트의 시간별 변화 과정을 기록하는 시스템이라고 볼 수 있다.

## 3. Git이란?

Git은 분산 버전 관리 시스템이다.

분산 버전 관리 시스템은 영어로 Distributed Version Control System이라고 한다.

Git은 소스 코드의 변경 이력을 관리하고, 특정 시점의 상태를 저장하며, 필요할 때 이전 상태로 되돌릴 수 있도록 도와주는 도구이다.

Git의 주요 역할은 다음과 같다.

| 역할 | 설명 |
|---|---|
| 변경 이력 관리 | 코드가 언제, 어떻게 수정되었는지 기록 |
| 버전 저장 | 특정 시점의 프로젝트 상태를 저장 |
| 롤백 | 문제가 생겼을 때 이전 버전으로 되돌림 |
| 브랜치 관리 | 여러 작업 흐름을 분리해서 관리 |
| 협업 지원 | 여러 개발자의 변경 사항을 병합 |

Git은 로컬 환경에서 동작한다.

즉, 인터넷이 없어도 내 컴퓨터에서 프로젝트의 변경 이력을 관리할 수 있다.

## 4. Git을 시간 여행으로 이해하기

Git은 프로젝트의 상태를 특정 시점으로 되돌릴 수 있다.

이 점에서 Git은 시간 여행과 비슷하게 이해할 수 있다.

예를 들어 정상적으로 동작하던 코드가 있었는데, 이후 코드를 수정하면서 오류가 발생했다고 가정할 수 있다.

Git을 사용하면 문제가 발생하기 전의 상태를 확인하거나, 필요한 경우 이전 버전으로 되돌릴 수 있다.

흐름은 다음과 같다.

    정상 동작하는 코드 저장
    → 코드 수정
    → 오류 발생
    → 이전 버전 확인
    → 정상 상태로 복구

즉, Git은 프로젝트의 과거 상태를 기록하고 다시 꺼내올 수 있게 해준다.

## 5. Git을 공간 여행으로 이해하기

Git은 브랜치를 통해 여러 작업 흐름을 나누어 관리할 수 있다.

이 점에서 Git은 공간 여행과도 비슷하게 이해할 수 있다.

예를 들어 하나의 프로젝트 안에서 여러 기능을 동시에 개발한다고 가정할 수 있다.

    main branch
    ├── login branch
    ├── board branch
    └── payment branch

각 기능을 별도의 브랜치에서 작업하면, 서로 다른 작업이 섞이지 않도록 관리할 수 있다.

로그인 기능을 개발하는 동안 게시글 기능이나 결제 기능에 영향을 주지 않고 독립적으로 작업할 수 있다.

즉, Git은 하나의 프로젝트 안에서 여러 작업 공간을 만들고 전환할 수 있게 해준다.

## 6. Git이 협업에서 중요한 이유

실제 개발 프로젝트는 혼자서 완성하는 경우보다 여러 사람이 함께 만드는 경우가 많다.

협업 과정에서는 다음과 같은 상황이 자주 발생한다.

- 여러 명이 같은 파일을 수정한다.
- 각자 다른 기능을 동시에 개발한다.
- 코드 변경 사항을 서로 공유해야 한다.
- 잘못된 코드가 병합될 수 있다.
- 이전 버전으로 되돌려야 할 수 있다.

Git은 이러한 협업 상황에서 코드의 변경 이력을 관리하고, 여러 개발자의 작업을 병합할 수 있도록 도와준다.

특히 브랜치를 사용하면 각 개발자가 독립적인 작업 공간에서 기능을 개발하고, 이후 안정적으로 합칠 수 있다.

## 7. Git과 GitHub의 차이

Git과 GitHub는 서로 다른 개념이다.

| 구분 | Git | GitHub |
|---|---|---|
| 개념 | 버전 관리 도구 | Git 저장소를 온라인에서 관리하는 플랫폼 |
| 동작 위치 | 로컬 환경 | 원격 서버 |
| 인터넷 필요 여부 | 필요 없음 | 필요함 |
| 주요 목적 | 코드 변경 이력 관리 | 코드 공유와 협업 |
| 예시 기능 | commit, branch, log | repository, issue, pull request |

Git은 내 컴퓨터에서 프로젝트의 변경 이력을 관리하는 도구이다.

GitHub는 Git으로 관리되는 코드를 온라인 저장소에 올리고, 다른 사람들과 협업할 수 있도록 도와주는 플랫폼이다.

정리하면 다음과 같다.

    Git
    → 로컬에서 버전 관리

    GitHub
    → 온라인에서 코드 공유 및 협업

## 8. GitHub가 필요한 이유

Git은 로컬 환경에서 사용할 수 있다.

하지만 여러 사람이 함께 작업하려면 코드를 공유할 수 있는 온라인 저장소가 필요하다.

GitHub는 Git 저장소를 온라인에서 호스팅하고 협업할 수 있도록 도와주는 플랫폼이다.

GitHub를 사용하면 다음과 같은 작업을 할 수 있다.

| 기능 | 설명 |
|---|---|
| Repository | 프로젝트 저장소 생성 |
| Push | 로컬 코드를 원격 저장소에 업로드 |
| Pull | 원격 저장소의 코드를 로컬로 가져오기 |
| Fork | 다른 사람의 저장소를 내 계정으로 복사 |
| Pull Request | 변경 사항을 원본 저장소에 병합 요청 |
| Issue | 버그, 개선 사항, 작업 내용 관리 |
| Actions | 빌드, 테스트, 배포 자동화 |

즉, GitHub는 Git을 기반으로 한 협업 플랫폼이다.

## 9. CLI와 GUI

Git을 사용하는 방식은 크게 CLI 방식과 GUI 방식으로 나눌 수 있다.

| 방식 | 설명 | 예시 |
|---|---|---|
| CLI | 명령어를 입력해서 사용하는 방식 | Git Bash, 터미널 |
| GUI | 그래픽 화면에서 사용하는 방식 | SourceTree, GitHub Desktop |

CLI는 Command Line Interface의 약자이다.

터미널에 명령어를 직접 입력해서 Git을 사용하는 방식이다.

GUI는 Graphical User Interface의 약자이다.

버튼, 메뉴, 그래프 화면 등을 통해 Git을 사용하는 방식이다.

Git의 개념을 정확히 이해하기 위해서는 CLI 명령어의 의미를 먼저 익히는 것이 좋다.

GUI 도구는 Git의 흐름을 시각적으로 확인할 때 도움이 된다.

## 10. SourceTree

SourceTree는 Git을 GUI 방식으로 사용할 수 있게 도와주는 도구이다.

SourceTree를 사용하면 Git의 커밋 이력과 브랜치 흐름을 그래프 형태로 확인할 수 있다.

SourceTree의 장점은 다음과 같다.

- Git 이력을 시각적으로 확인할 수 있다.
- 브랜치 구조를 보기 쉽다.
- 변경된 파일을 화면에서 확인할 수 있다.
- CLI 명령어가 익숙하지 않아도 Git 흐름을 파악할 수 있다.

하지만 Git의 기본 원리를 이해하기 위해서는 CLI 방식의 기본 명령어를 먼저 학습하는 것이 중요하다.

## 11. Git 최초 설정

Git을 처음 사용할 때는 사용자 이름과 이메일을 설정해야 한다.

이 정보는 커밋을 남길 때 작성자 정보로 사용된다.

사용자 이름 설정 명령어는 다음과 같다.

    git config --global user.name "사용자이름"

사용자 이메일 설정 명령어는 다음과 같다.

    git config --global user.email "이메일주소"

설정된 사용자 이름과 이메일을 확인하려면 다음 명령어를 사용할 수 있다.

    git config --global user.name
    git config --global user.email

이 설정은 GitHub 계정과 완전히 같은 개념은 아니지만, 보통 GitHub에 사용하는 이름과 이메일을 맞춰 설정한다.

## 12. 기본 브랜치 이름 설정

예전에는 Git의 기본 브랜치 이름으로 master가 많이 사용되었다.

하지만 최근에는 기본 브랜치 이름으로 main을 많이 사용한다.

새 저장소를 만들 때 기본 브랜치 이름을 main으로 설정하려면 다음 명령어를 사용할 수 있다.

    git config --global init.defaultBranch main

이 설정을 해두면 이후 git init으로 새 저장소를 만들 때 기본 브랜치가 main으로 생성된다.

## 13. 버전 관리의 핵심 흐름

Git을 이용한 버전 관리의 기본 흐름은 다음과 같다.

    파일 생성 또는 수정
    → 변경 상태 확인
    → 변경 파일을 스테이징
    → 커밋으로 버전 기록
    → 커밋 이력 확인

각 단계는 다음 명령어와 연결된다.

| 단계 | 명령어 | 설명 |
|---|---|---|
| 상태 확인 | git status | 현재 변경 상태 확인 |
| 스테이징 | git add | 변경 파일을 커밋 준비 영역에 추가 |
| 버전 기록 | git commit | 변경 사항을 하나의 버전으로 저장 |
| 이력 확인 | git log | 커밋 기록 확인 |

Git에서는 파일을 수정했다고 바로 버전으로 저장되지 않는다.

먼저 변경 파일을 스테이징 영역에 올리고, 이후 커밋을 통해 하나의 버전으로 기록한다.

## 14. 정리

Git은 소스 코드의 변경 이력을 관리하기 위한 분산 버전 관리 시스템이다.

Git을 사용하면 특정 시점의 프로젝트 상태를 기록하고, 문제가 발생했을 때 이전 상태로 되돌릴 수 있다.

또한 브랜치를 통해 여러 작업 흐름을 분리하여 관리할 수 있고, 협업 과정에서 발생하는 코드 변경 사항을 체계적으로 관리할 수 있다.

GitHub는 Git 저장소를 온라인에서 관리하고 협업할 수 있게 해주는 플랫폼이다.

따라서 Git과 GitHub는 구분해서 이해해야 한다.

핵심 개념은 다음과 같다.

- 버전 관리는 파일의 변경 이력을 기록하고 관리하는 것이다.
- Git은 분산 버전 관리 시스템이다.
- Git은 로컬 환경에서 동작한다.
- GitHub는 온라인 저장소와 협업 기능을 제공하는 플랫폼이다.
- Git은 인터넷이 없어도 사용할 수 있다.
- GitHub는 원격 저장소를 사용하므로 인터넷이 필요하다.
- Git은 시간 여행처럼 이전 버전으로 되돌아갈 수 있다.
- Git은 브랜치를 통해 여러 작업 공간을 나누어 관리할 수 있다.
- CLI는 명령어 기반 사용 방식이다.
- GUI는 그래픽 화면 기반 사용 방식이다.
- Git을 처음 사용할 때는 사용자 이름과 이메일을 설정해야 한다.
